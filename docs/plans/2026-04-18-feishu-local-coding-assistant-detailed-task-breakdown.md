# 飞书本地聊天式代码助手细化开发任务清单

- 文档日期：2026-04-18
- 文档状态：可执行任务拆解
- 来源文档：[2026-04-18-feishu-local-coding-assistant-implementation-plan.md](./2026-04-18-feishu-local-coding-assistant-implementation-plan.md)
- 适用范围：本地电脑部署的飞书聊天式代码助手

## 1. 使用方式

本清单用于把实施文档中的 `M1` 到 `M4` 再下钻到“可排期、可分配、可验收”的开发任务级别。任务默认按文档顺序推进；若主链路任务未完成，不建议跨阶段并行开发核心功能。

## 2. 拆解约定

- 编号格式使用 `M{阶段}-{序号}`，例如 `M2-07`。
- 每个任务默认都要补齐对应的代码、脚本、SQL、配置或文档，不接受“只写代码不收口”。
- 每个任务只有在以下条件都满足后才能勾选完成：
  - 代码或文档已落盘到仓库。
  - 至少有一条可重复执行的验证动作。
  - 失败路径具备可定位的日志或数据库状态。
  - 若影响启动、配置或运维方式，`README` 或运行文档已同步。

## 3. M1：最小可用链路

### 3.1 工程与配置

- [ ] `M1-01` 初始化 Bun + TypeScript 工程骨架：创建 `package.json`、`tsconfig.json`、基础脚本命令；完成标准：依赖可安装，存在 `dev`、`start`、`typecheck` 之类的最小命令入口。
- [ ] `M1-02` 建立源码目录和运行目录约定：创建 `src/apps`、`src/core`、`src/domains`、`src/infra` 目录，并明确 `%LOCALAPPDATA%\\FeishuCodexBot\\` 下的数据、日志路径。
- [ ] `M1-03` 实现统一配置加载模块：支持环境变量读取 `FEISHU_APP_ID`、`FEISHU_APP_SECRET`、`OPENAI_API_KEY`、数据库路径、日志路径；完成标准：启动时可打印脱敏后的有效配置摘要。
- [ ] `M1-04` 提供 `.env.example` 和最小 `README.md`：列出必需变量、启动步骤、测试前置条件；完成标准：新环境按文档可完成本地启动。
- [ ] `M1-05` 落地基础日志模块：支持 `gateway` 与 `worker` 的进程级日志输出；完成标准：启动、停止、异常退出均有明确日志。

### 3.2 数据库与基础模型

- [ ] `M1-06` 实现 SQLite 连接工厂与 `WAL` 初始化：封装数据库打开、路径创建、连接复用和启动时 `PRAGMA` 设置；依赖：`M1-02`。
- [ ] `M1-07` 编写 migration runner：支持按顺序执行 SQL 文件，并记录已执行版本；完成标准：重复执行初始化不会报错。
- [ ] `M1-08` 编写 `migrations/0001_init.sql`：创建 `raw_events`、`conversations`、`messages`、`jobs` 四张基础表及首批必要索引。
- [ ] `M1-09` 定义领域模型与仓储接口：明确事件、会话、消息、任务的 TypeScript 类型和基础读写接口，避免网关和 worker 直接散落 SQL。

### 3.3 飞书接入与消息落库

- [ ] `M1-10` 封装飞书鉴权客户端：实现 tenant access token 获取、缓存与失效刷新；完成标准：接口调用前可以稳定拿到可用 token。
- [ ] `M1-11` 建立飞书长连接接入：拉起长连接客户端并监听消息事件；完成标准：收到真实测试消息时进程日志可见事件进入。
- [ ] `M1-12` 实现消息事件解析与过滤：仅接受文本消息，忽略机器人自身消息和当前阶段不支持的事件类型。
- [ ] `M1-13` 设计内部消息 DTO 和 conversation key 规则：统一飞书事件到内部结构的映射，明确单聊会话键生成方式。
- [ ] `M1-14` 实现 gateway 的入库与入队事务：把原始事件写入 `raw_events`，创建或更新 `conversations`，写入用户消息，并创建 `jobs(status=queued)`。

### 3.4 worker 生成闭环

- [ ] `M1-15` 实现 worker 的最小任务领取逻辑：轮询 `queued` 任务，领取后改成 `running`，失败时能回写错误原因。
- [ ] `M1-16` 实现最小上下文读取：从 `messages` 拉取当前用户消息和最近必要上下文，先不做复杂摘要与追问策略。
- [ ] `M1-17` 封装 OpenAI Responses API 客户端：输入内部 DTO，输出标准化 assistant 回复；完成标准：能稳定发起一次最小请求并拿到文本回复。
- [ ] `M1-18` 实现 assistant 回复持久化与飞书回写：把模型回复写入 `messages`，调用飞书 reply 接口回写，并把任务改成 `succeeded`。
- [ ] `M1-19` 补齐最小错误处理链路：OpenAI 失败、飞书回复失败、数据库异常时，任务状态和日志能反映失败点。

### 3.5 联调与收口

- [ ] `M1-20` 执行单聊端到端联调：验证“飞书发文本 -> gateway 入队 -> worker 生成 -> 飞书回消息”闭环至少可重复通过 3 次。
- [ ] `M1-21` 核对数据库状态：确认 `raw_events`、`conversations`、`messages`、`jobs` 均能反映一次完整交互。
- [ ] `M1-22` 核对日志可排障性：确认主链路各关键节点都有日志，且失败时能从日志直接定位到阶段内问题。

## 4. M2：稳定性增强

### 4.1 幂等与任务调度

- [ ] `M2-01` 为 `raw_events(platform, event_id)` 和 `messages(platform, platform_message_id)` 增加唯一约束并补 migration。
- [ ] `M2-02` 在 gateway 中实现重复事件短路逻辑：同一飞书事件重复投递时不重复创建消息和任务。
- [ ] `M2-03` 扩展 `jobs` 表结构：增加 `locked_by`、`lease_expires_at`、`attempt_count`、`available_at` 等字段。
- [ ] `M2-04` 重写 worker 抢占逻辑为事务领取：多 worker 场景下同一任务只能被一个实例成功领取。
- [ ] `M2-05` 增加租约续租机制：长耗时任务执行期间定期刷新租约，避免误回收。
- [ ] `M2-06` 增加僵尸任务回收逻辑：worker 异常退出后，过期租约任务可重新回到可执行状态。

### 4.2 重试与恢复点

- [ ] `M2-07` 建立错误分类模型：区分可重试、不可重试、可降级错误，并给出统一错误码或错误类型。
- [ ] `M2-08` 新增 `job_attempts` 记录：保存每次尝试的开始时间、结束时间、错误类型和错误摘要。
- [ ] `M2-09` 实现指数退避与重试调度：依赖 `available_at` 控制下次可执行时间，避免快速打爆外部接口。
- [ ] `M2-10` 为 OpenAI `429`、`5xx`、超时建立自动重试策略，并限制最大重试次数。
- [ ] `M2-11` 新增 `deliveries` 表：把“生成回答”和“回写飞书”拆成两个可恢复步骤。
- [ ] `M2-12` 实现 delivery-only 重试：飞书发送失败时只重试投递，不重复调用 OpenAI 生成回答。

### 4.3 群聊边界与观测

- [ ] `M2-13` 实现群聊 `@机器人` 检测：只有明确被提及时才继续走主链路。
- [ ] `M2-14` 加入允许范围控制：支持 `ALLOW_GROUPS`、`ALLOWED_CHAT_IDS`、`ALLOWED_USER_IDS` 等配置。
- [ ] `M2-15` 增加机器人自触发保护：避免机器人回复再次被自己消费形成循环。
- [ ] `M2-16` 升级为结构化日志：统一输出 `trace_id`、`job_id`、`conversation_id`、`attempt` 等字段。
- [ ] `M2-17` 输出 `gateway.health.json` 与 `worker.health.json`：包含最后成功时间、最近错误摘要、当前积压量等基础信息。

### 4.4 验证与回归

- [ ] `M2-18` 增加重复投递测试：验证同一飞书消息重复到达不会重复回复。
- [ ] `M2-19` 增加异常恢复测试：模拟 worker 崩溃、租约过期、OpenAI 临时失败、飞书投递失败等场景。
- [ ] `M2-20` 补充阶段故障排查记录：把常见失败路径、数据库表现和日志特征写入运维文档草稿。

## 5. M3：会话质量增强

### 5.1 会话续链

- [ ] `M3-01` 为 `conversations` 增加 `last_response_id`、`summary_text` 等字段，并补 migration。
- [ ] `M3-02` 在 OpenAI 成功响应后持久化 `response_id`，为后续续链提供数据基础。
- [ ] `M3-03` 调用 OpenAI 时优先使用 `previous_response_id`，减少重复传输上下文。
- [ ] `M3-04` 处理 `previous_response_id` 失效回退：当 OpenAI 拒绝续链时，自动退回到本地消息窗口 + 摘要。

### 5.2 上下文构建

- [ ] `M3-05` 设计本地历史窗口规则：定义最近 N 轮有效消息的选取方式、角色顺序和 token 预算。
- [ ] `M3-06` 实现噪音过滤：剔除寒暄、重复系统提示、无关回执，保留对代码问答有用的信息。
- [ ] `M3-07` 实现单聊与群聊上下文隔离：群聊要区分不同用户，避免不同人的问题串上下文。
- [ ] `M3-08` 设计摘要触发条件：按消息数、总长度或轮次数触发摘要生成。
- [ ] `M3-09` 实现摘要生成与持久化：把技术栈、关键报错、已尝试方案、未解决问题写入 `summary_text`。

### 5.3 提示词与回复策略

- [ ] `M3-10` 固化系统提示词：明确代码助手的行为边界、语言风格、不能伪造运行结果等约束。
- [ ] `M3-11` 建立问题分类策略：至少覆盖报错分析、代码解释、代码生成、方案建议、信息不足追问五类输入。
- [ ] `M3-12` 实现上下文不足追问逻辑：证据不足时优先补问关键信息，而不是直接给不可靠答案。
- [ ] `M3-13` 实现超长输入裁剪：针对日志、报错栈、代码块做分段保留和裁剪策略。
- [ ] `M3-14` 控制默认回复长度：保证飞书中默认回复可读、简洁，必要时提示用户继续展开。

### 5.4 质量验证

- [ ] `M3-15` 建立质量样本集：覆盖 Node 报错、SQL 问题、代码解释、架构问题、信息不足问题等典型场景。
- [ ] `M3-16` 为样本集定义验收口径：至少包含“是否连续上下文、是否先给结论、是否虚构信息、是否需要追问”等评分点。
- [ ] `M3-17` 执行多轮回归测试：验证多轮单聊连续性、群聊隔离、摘要回退和追问行为是否稳定。
- [ ] `M3-18` 根据样本结果调优摘要、裁剪和提示词：直到回答质量达到阶段可接受标准。

## 6. M4：运维与发布增强

### 6.1 配置与启动治理

- [ ] `M4-01` 梳理完整运行配置清单：明确必填、选填、默认值、敏感项和可热更新项。
- [ ] `M4-02` 引入本地配置文件 + 环境变量覆盖机制：例如 `config/default.json` 与本地覆盖文件的加载顺序。
- [ ] `M4-03` 在启动阶段加入配置校验：缺失关键密钥、路径不可写、数据库不可访问时应拒绝启动并输出可读错误。
- [ ] `M4-04` 增加 `schema_version` 或等效版本记录机制，并在启动时执行 migration 检查。

### 6.2 Windows 常驻运行

- [ ] `M4-05` 编写计划任务安装脚本：支持一次性安装 gateway 和 worker 的开机/登录启动项。
- [ ] `M4-06` 编写计划任务卸载和重装脚本：便于升级和回滚时清理旧任务。
- [ ] `M4-07` 配置失败重启策略：进程异常退出后可自动拉起，并控制最大重试频率。
- [ ] `M4-08` 增加单实例保护：避免重复启动多个 gateway 或多个 worker 造成竞争和重复消费。

### 6.3 备份、恢复与清理

- [ ] `M4-09` 编写数据库备份脚本：支持按时间戳输出备份文件，并记录备份结果。
- [ ] `M4-10` 约定备份保留策略：明确保留周期、数量上限和清理规则。
- [ ] `M4-11` 编写恢复操作说明或脚本：让本机可以从备份恢复到可运行状态。
- [ ] `M4-12` 实现数据清理任务：清理过旧 `raw_events`、历史日志、过期 `deliveries` 等数据。
- [ ] `M4-13` 增加活跃数据保护：清理任务不能误删活跃会话、未完成任务和必要审计记录。

### 6.4 文档与发布收口

- [ ] `M4-14` 编写 `docs/runbook.md`：覆盖启动、停止、重启、升级、备份、恢复的标准步骤。
- [ ] `M4-15` 编写 `docs/operations.md`：覆盖日志位置、health 文件说明、常见错误定位和处理方法。
- [ ] `M4-16` 约定版本发布与升级回滚流程：明确升级前检查、升级步骤、失败回滚和数据保护要求。
- [ ] `M4-17` 完成全链路演练：至少手动验证登录自启、升级、备份、恢复、清理五类场景。
- [ ] `M4-18` 冻结当前发布基线：确认没有阻塞长期运行的高优先级遗留项，再宣布 M4 完成。

## 7. 推荐推进方式

1. 先完成 `M1-01` 到 `M1-05`，把工程和启动基础打稳，再进入飞书和 OpenAI 接入。
2. `M2` 不要只做“重试”而跳过“租约恢复”和“投递解耦”，这三者必须一起形成完整恢复能力。
3. `M3` 的目标不是堆更多 prompt，而是先把会话数据、回退路径和质量样本固化。
4. `M4` 必须安排至少一次手动演练，不接受“脚本写好了但没跑过”的收尾方式。
