# 飞书本地聊天式代码助手分阶段实施文档

- 文档日期：2026-04-18
- 文档状态：可执行实施计划
- 关联设计文档：[2026-04-18-feishu-local-coding-assistant-design.md](./2026-04-18-feishu-local-coding-assistant-design.md)
- 细化任务清单：[2026-04-18-feishu-local-coding-assistant-detailed-task-breakdown.md](./2026-04-18-feishu-local-coding-assistant-detailed-task-breakdown.md)
- 适用范围：本地电脑部署的飞书聊天式代码助手
- 目标技术栈：Node.js 24 + TypeScript + Bun + SQLite

## 1. 文档目的

本文件用于把已确认的详细设计转化为可执行的实施计划。设计文档负责回答“要做什么”和“为什么这样做”，本文件负责回答“按什么顺序做、每一阶段产出什么、如何验收、哪些风险需要前置控制”。

本实施文档不重新讨论已经确认的关键设计决策，包括：

- 飞书接入方式使用长连接
- 本地双进程架构：`bot-gateway` + `assistant-worker`
- OpenAI 接口使用 `Responses API`
- 本地持久化使用 `SQLite`
- 默认模型使用 `gpt-5.4-mini`

## 2. 实施原则

### 2.1 先打通链路，再补强可靠性

第一目标不是一次性做完整系统，而是尽快建立“飞书收消息 -> 本地入队 -> worker 生成 -> 飞书回消息”的可验证主链路。

### 2.2 每个阶段都必须可独立验收

每个阶段都需要明确：

- 进入条件
- 主要工作包
- 产出物
- 验收标准
- 主要风险

未达到出口条件，不进入下一阶段。

### 2.3 不重开架构决策

实施过程只允许在不改变总体设计前提下细化模块和接口。涉及架构重开的问题必须单独记录，不在执行中直接漂移。

### 2.4 以本地可维护性为第一优先级

该系统运行在本机而非云环境，因此要优先保障：

- 可启动
- 可恢复
- 可排障
- 可备份

而不是优先追求复杂平台能力。

## 3. 总体阶段划分

| 阶段 | 名称 | 核心目标 | 结果定位 |
|------|------|----------|----------|
| M1 | 最小可用链路 | 打通从飞书到 OpenAI 再回飞书的基本路径 | 能跑 |
| M2 | 稳定性增强 | 建立去重、重试、恢复与观测能力 | 可靠 |
| M3 | 会话质量增强 | 提升上下文连续性和答复质量 | 好用 |
| M4 | 运维与发布增强 | 完成启动、升级、备份、清理和运行文档 | 可长期运行 |

建议严格按 `M1 -> M2 -> M3 -> M4` 顺序执行，不并行跨阶段开发主链路功能。

## 4. 阶段前置准备

在进入 M1 前，先完成以下基础准备：

1. 初始化仓库基础结构。
2. 确定运行目录约定：
   - 源码目录在仓库内
   - 数据与日志目录在 `%LOCALAPPDATA%\\FeishuCodexBot\\`
3. 准备本地密钥注入方式：
   - 开发期环境变量
   - 后续支持 Windows Credential Manager
4. 确定飞书测试对象：
   - 测试机器人
   - 测试群
   - 测试单聊账号
5. 创建基础文档占位：
   - `README.md`
   - `.env.example`
   - `docs/runbook.md`

## 5. M1：最小可用链路

### 5.1 阶段目标

建立从飞书文本消息到 OpenAI 回复再回飞书的最小闭环，优先支持单聊文本场景。

### 5.2 进入条件

- 设计文档已确认
- 飞书企业自建应用已创建并具备长连接接收事件能力
- 已拿到可用的 `FEISHU_APP_ID`、`FEISHU_APP_SECRET`、`OPENAI_API_KEY`
- 本机可运行 `bun`、`node`

### 5.3 主要工作包

#### WP1：项目脚手架与工程基础

- 创建 `package.json`
- 创建 `tsconfig.json`
- 创建 Bun 脚本命令
- 建立源码目录结构：
  - `src/apps/bot-gateway`
  - `src/apps/assistant-worker`
  - `src/core`
  - `src/domains`
  - `src/infra`
- 引入基础依赖：
  - 配置加载
  - SQLite 驱动
  - 日志库
  - OpenAI SDK
  - 飞书 SDK 或 HTTP 客户端封装

#### WP2：配置与启动骨架

- 实现统一配置加载模块
- 支持从环境变量读取密钥
- 提供 `.env.example`
- 实现 gateway 和 worker 的独立启动入口

#### WP3：SQLite 初始化

- 创建数据库连接模块
- 创建 migration 执行入口
- 初始化基础表：
  - `raw_events`
  - `conversations`
  - `messages`
  - `jobs`
- 打开 `WAL`

#### WP4：飞书长连接接入

- 建立与飞书的长连接
- 监听接收消息事件
- 仅处理文本消息
- 忽略机器人自己的消息
- 单聊优先，不要求先支持群聊复杂策略

#### WP5：消息规范化与入队

- 设计内部消息 DTO
- 将飞书事件落入 `raw_events`
- 创建或更新 `conversations`
- 将用户消息写入 `messages`
- 创建 `jobs(status=queued)`

#### WP6：worker 最小处理链路

- 轮询领取 `queued` 任务
- 将任务置为 `running`
- 读取当前消息和最小上下文
- 调用 OpenAI Responses API
- 将 assistant 回复写入 `messages`
- 调用飞书消息回复接口回写
- 成功后任务改为 `succeeded`

#### WP7：基础日志与错误处理

- 为 gateway 和 worker 增加基础日志
- 记录启动、入队、出队、OpenAI 调用成功/失败、飞书发送成功/失败
- 未捕获异常时输出明确错误日志

### 5.4 建议交付物

- `package.json`
- `tsconfig.json`
- `.env.example`
- `src/apps/bot-gateway/*`
- `src/apps/assistant-worker/*`
- `src/core/config/*`
- `src/core/logger/*`
- `src/core/db/*`
- `src/domains/feishu/*`
- `src/domains/openai/*`
- `migrations/0001_init.sql`
- 最小 `README.md`

### 5.5 验收标准

- 单聊向机器人发送一条文本代码问题，机器人可正常回复
- SQLite 中可看到事件、消息、会话、任务记录
- worker 可从队列取任务并完成处理
- 日志中可看到完整主链路
- 普通问题端到端闭环可复现

### 5.6 阶段风险

- 飞书长连接 SDK 行为与预期不一致
- 本地环境变量或密钥配置错误
- OpenAI 调用成功但飞书回消息接口参数不匹配
- schema 设计过早冻结导致后续变更代价偏高

### 5.7 出口条件

满足以下全部条件才进入 M2：

- 单聊文本主链路稳定可复现
- gateway 与 worker 可独立启动
- 数据库初始化与 migration 可重复执行
- 关键日志可支撑基本排障

## 6. M2：稳定性增强

### 6.1 阶段目标

在已打通主链路的基础上，建立系统可靠性能力，包括去重、重试、租约恢复、回写解耦、群聊触发控制和基础健康信息。

### 6.2 进入条件

- M1 已通过
- 已识别出主链路上的关键失败点
- 基础 schema 和模块边界基本稳定

### 6.3 主要工作包

#### WP1：去重与幂等

- 为 `raw_events(platform, event_id)` 增加唯一约束
- 为 `messages(platform, platform_message_id)` 增加唯一约束
- 在 gateway 中实现幂等入队逻辑
- 验证飞书重复投递时不会重复回复

#### WP2：租约锁与任务恢复

- 为 `jobs` 增加：
  - `locked_by`
  - `lease_expires_at`
  - `attempt_count`
  - `available_at`
- worker 采用事务抢占任务
- 增加续租机制
- 增加僵尸任务回收逻辑

#### WP3：重试与错误分类

- 区分可重试、不可重试、可降级错误
- 实现指数退避
- 记录 `job_attempts`
- 为 OpenAI 429、5xx、超时建立自动重试

#### WP4：生成与投递解耦

- 新增 `deliveries` 表
- 将“生成回答”和“回写飞书”拆成两个恢复点
- 飞书发送失败时只重试投递，不重复生成

#### WP5：群聊触发控制

- 支持群聊只对 `@机器人` 响应
- 支持 `ALLOW_GROUPS`、`ALLOWED_CHAT_IDS`、`ALLOWED_USER_IDS`
- 避免机器人自触发循环

#### WP6：健康状态与结构化日志

- 输出 `gateway.health.json`
- 输出 `worker.health.json`
- 日志升级为结构化格式
- 记录 `trace_id`、`job_id`、`conversation_id`、`attempt`

### 6.4 建议交付物

- `migrations/0002_reliability.sql`
- `src/domains/jobs/*`
- `src/domains/messages/idempotency/*`
- `src/core/health/*`
- `src/core/errors/*`
- 群聊触发策略配置项
- 失败重试测试样例

### 6.5 验收标准

- 同一飞书消息重复投递不重复回复
- OpenAI 临时失败时任务能自动重试
- worker 异常退出后，租约过期任务可恢复
- 飞书发送失败时不会重复调用 OpenAI
- 群聊未 `@机器人` 时不回复
- health 文件持续更新

### 6.6 阶段风险

- SQLite 锁竞争导致 worker 任务领取抖动
- 错误分类不准导致不该重试的任务反复重试
- 重试与投递拆分后状态流转复杂度提升

### 6.7 出口条件

- 幂等、重试、租约恢复都经测试验证
- 群聊触发边界明确且可控
- 失败路径可通过日志和数据库状态定位

## 7. M3：会话质量增强

### 7.1 阶段目标

提升聊天式代码助手的上下文连续性、答复相关性和行为一致性，使其从“能回答”提升到“像一个可靠的代码助手”。

### 7.2 进入条件

- M2 已通过
- 主链路和恢复能力足够稳定
- 已具备基本日志和会话持久化

### 7.3 主要工作包

#### WP1：`previous_response_id` 会话续链

- 在 `conversations` 中维护 `last_response_id`
- OpenAI 调用优先使用 `previous_response_id`
- 对失效场景做回退处理

#### WP2：本地历史窗口

- 从 `messages` 中读取最近 N 轮有效消息
- 丢弃寒暄和无关系统内容
- 保证单聊和群聊的上下文隔离

#### WP3：会话摘要机制

- 增加 `summary_text`
- 在消息数量达到阈值后生成摘要
- 摘要保留：
  - 技术栈
  - 关键报错
  - 已尝试方案
  - 当前未解决问题

#### WP4：提示词与追问策略

- 固定系统提示词
- 区分回答类型：
  - 报错分析
  - 代码解释
  - 代码生成
  - 方案建议
  - 信息不足追问
- 当上下文不足时优先追问，而不是强行回答

#### WP5：输入裁剪与回复长度控制

- 对超长日志和代码做长度限制
- 引导用户贴最小必要片段
- 控制默认回复长度适配飞书 IM

#### WP6：质量验证样例

- 建立一组典型问题样本：
  - Node 报错
  - SQL 问题
  - 代码解释
  - 架构问题
  - 信息不足问题
- 校验回复是否符合“结论 -> 原因 -> 示例”的输出约束

### 7.4 建议交付物

- `src/domains/conversation/*`
- `src/domains/openai/prompting/*`
- `src/domains/openai/context-builder/*`
- `src/domains/openai/response-policy/*`
- 会话摘要生成逻辑
- 质量样例文档或测试集

### 7.5 验收标准

- 多轮单聊可保持连续上下文
- 群聊中不同用户不会串上下文
- `previous_response_id` 无效时可退回本地消息窗口 + 摘要
- 上下文不足时会追问关键问题
- 默认回复足够简洁，不虚构项目结构或运行结果

### 7.6 阶段风险

- 会话摘要质量不足导致上下文漂移
- 过度裁剪影响回答质量
- 提示词约束太强导致回答机械化

### 7.7 出口条件

- 多轮对话质量达到可接受水平
- 失败回退路径覆盖完毕
- 回答风格和行为边界稳定

## 8. M4：运维与发布增强

### 8.1 阶段目标

把系统从“开发环境可运行服务”提升为“本机可长期运行服务”，补齐启动、升级、备份、清理与运行文档。

### 8.2 进入条件

- M3 已通过
- 功能与质量基本稳定
- 不再频繁调整数据库主结构

### 8.3 主要工作包

#### WP1：Windows 计划任务启动

- 编写安装计划任务脚本
- 支持登录后自动拉起 gateway 和 worker
- 支持失败重启策略

#### WP2：配置文件规范化

- 梳理运行所需配置项
- 支持本地配置文件 + 环境变量覆盖
- 规范 `config/default.json` 或等效配置目录

#### WP3：数据库 migration 流程

- 增加 `schema_version`
- 启动时执行 migration 检查
- migration 失败则拒绝启动

#### WP4：备份与恢复脚本

- 增加数据库备份脚本
- 约定备份保留策略
- 提供恢复操作说明

#### WP5：数据清理与维护任务

- 清理过旧的 `raw_events`
- 清理历史日志或过期 delivery 记录
- 保留必要的审计和摘要信息

#### WP6：运行与排障文档

- 编写启动说明
- 编写升级说明
- 编写常见故障排查步骤
- 编写日志与 health 文件使用说明

### 8.4 建议交付物

- `scripts/install-scheduled-tasks.ps1`
- `scripts/backup-db.ps1`
- `scripts/cleanup-db.ts`
- `migrations/*`
- `docs/runbook.md`
- `docs/operations.md`

### 8.5 验收标准

- 登录后可自动启动服务
- 升级流程可执行且可回滚
- 备份与恢复流程经手动演练通过
- 清理任务不会破坏活跃会话
- 运维文档足以指导本机排障

### 8.6 阶段风险

- 计划任务配置错误导致开机不启动或重复启动
- migration 管理不当导致本地数据库损坏
- 清理策略误删活跃数据

### 8.7 出口条件

- 本机启动、停止、升级、备份、恢复全链路可操作
- 运维文档完整
- 系统具备长期运行的基本条件

## 9. 跨阶段测试与验证要求

### 9.1 每阶段必须补齐的测试

- 单元测试：纯逻辑模块
- 集成测试：数据库状态流转、任务处理、重试恢复
- 端到端测试：飞书消息到最终回复

### 9.2 关键回归项

以下项在每阶段后都应回归验证：

- 单聊消息是否可正常回复
- 群聊 `@机器人` 逻辑是否正常
- 数据库 migration 是否可重复执行
- 日志是否保留关键 trace 信息
- 重启后历史会话是否仍可读取

## 10. 建议的实施顺序与里程碑判定

### 10.1 实施顺序

1. M1 完成工程骨架与主链路
2. M2 补齐幂等、恢复、限流与观测
3. M3 优化上下文和答复质量
4. M4 固化本机运维与发布能力

### 10.2 里程碑判定规则

只有满足该阶段所有出口条件，且关键回归项通过，才允许进入下一阶段。任何会影响上阶段核心目标的问题，必须先回补，不允许带病推进。

## 11. 暂缓项

以下内容明确暂缓，不进入当前实施计划：

- 仓库自动读取与本地代码执行
- 文件上传与附件解析
- GUI 面板
- 向量检索或知识库
- 多机器人实例管理
- 云端部署形态

## 12. 实施完成定义

当以下条件全部满足时，可认为该项目达到当前设计范围内的实施完成状态：

- M1 到 M4 全部通过出口条件
- 关键回归项稳定通过
- 本地运行、重启、升级、备份、恢复都有明确路径
- 用户能在飞书中稳定使用聊天式代码助手完成代码问答场景

届时，如需继续扩展能力，应以新的设计文档或变更提案进入下一轮规划，而不是直接在现有实施计划中追加范围。
